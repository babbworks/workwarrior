# Built for Agents Too

There's a version of the future where the person checking off tasks isn't always a person. AI agents are already managing workflows, filing pull requests, reviewing code, sending messages, and making decisions that used to require a human hand. Most productivity infrastructure wasn't designed for this. It was designed for a person with eyes and a mouse, navigating a web UI that assumes you're reading it. The APIs are bolted on. The data models are optimized for display. The concepts — "inbox," "dashboard," "notification" — are human concepts, not machine concepts.

Workwarrior was designed differently. Not because it anticipated AI agents specifically, but because it was designed for the terminal — and the terminal has always been a place where programs talk to other programs. Commands have structured outputs. Queries return parseable data. Exit codes mean something. When an AI agent needs to manage its own work, the same interface a human uses works equally well: `ww next`, `ww start`, `ww log`, `ww done`. No scraping. No API keys. No rate limits. Just a CLI that does what it says.

An agent using Workwarrior gets its own profile. Its task queue, its time log, its journal entries, its financial records — all isolated, all labeled, all searchable, all exportable. The agent can log what it did, why it did it, how long it took, and what it cost, using the same commands a human would use. When you go back to audit what the agent did last Tuesday, the record is there — not in a vendor's log system, not in a cloud database you don't control, but in plaintext files on your machine, under your directory, owned by you.

This matters because the value of a productivity system isn't just what you do with it today. It's what you can learn from it tomorrow. When your agent logs a journal entry saying "resolved ambiguity in the payment API design by choosing option B — option A would have required a breaking change in three downstream services," that sentence is preserved in a format that will be readable in twenty years. It's not trapped in a system's proprietary storage. It's a line in a file.

The profile model extends naturally to multi-agent setups. You might have one profile for your personal tasks, one for your work projects, one for the AI agent that manages your GitHub issues, and one for the agent that handles your recurring operational tasks. Each profile is sealed. The agents don't see each other's queues. They can't accidentally log time to the wrong project or mix up which tasks belong to which context. The isolation that protects a human from context bleed protects agents from it too.

Workwarrior doesn't try to be the AI layer. There are better tools for inference, planning, and memory. What it provides is the operational substrate: a place for an agent to have a task queue, a clock, a journal, and a ledger — all of it local, all of it auditable, all of it in formats that last. The agent brings the intelligence. Workwarrior gives it somewhere to work.
